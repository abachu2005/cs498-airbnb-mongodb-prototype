Collection layout

Time-series data is stored separately from listings.

Each of the four InsideAirbnb CSVs maps to one MongoDB collection. Calendar and review rows are kept out of the listing document so that adding a single booking or review does not require rewriting a large embedded array.

Demonstration

Q2 · anti-join over neighborhoods

Identify neighborhoods with no available listings in a chosen month.

  • 1. $match calendar by city, month, and available:true; $group by listing
  • 2. $lookup listings, project neighborhood, take the distinct active set
  • 3. Anti-join in application code against the full neighborhoods list for that city
Full-data result for Portland in 2026-06: 2 of 95 neighborhoods (Maywood Park, Woodland Park) had no available listings.

Query-plan note · Q2
Three indexed reads, no full collection scans. The $match uses calendar { city, date, available }; the $lookup uses listings { city, listing_id }; the anti-join runs in application code over a small set (~95 neighborhood strings). 07Q3 — Salem entire-home windows
Demonstration

Q3 · minimum-nights-aware availability scan

Compute bookable windows that respect each listing's minimum_nights.

  • 1. One indexed find for Salem entire-home listings
  • 2. One sorted find over calendar { city, listing_id, date }
  • 3. Walk consecutive available:true runs in application code; keep day d only if remaining-run-length is at least minimum_nights[d]
A listing may report available on a Saturday while requiring a seven-night minimum. If only Saturday and Sunday are open the listing is not actually bookable, and this condition cannot be expressed inside an aggregation pipeline alone.
2 indexed reads per listing minimum-nights aware
localhost:4173 / Q3 — Salem availability windows
Demo UI · full data · 151 Salem entire-home listings · 2,330 bookable nights total in 2026-06
Query-plan note · Q3
Two indexed reads per listing produce a predictable, bounded footprint. The interval-detection logic runs in application code rather than in aggregation; this is a deliberate trade — clarity and minimum-nights awareness in exchange for one additional round trip per listing.

11Critique
Design and query-plan review

What we would change, ranked by impact

The 4-collection, 8-index core held up under the full data load.

01 · Index key order

host_id_1_city_1 has the wrong key order.

Should be { city, host_id } so it shares a city prefix with the other three listings indexes. This affects Q6's same-host lookup.

02 · Date-as-string rationale

Correct decision; original justification was incomplete.

Stage 2 justified it primarily on storage. The larger benefit was query simplicity — Q5's regex and $substr avoid any $toDate conversion.

03 · Embed neighborhoods

A per-city neighborhood array would replace the $lookup.

Removes the $lookup from Q2 entirely, at the cost of a few hundred bytes per city, in exchange for one fewer cross-collection hop.

04 · Query-plan bottleneck

Q5 performs a COLLSCAN over reviews.

All 3.2 M reviews are scanned to regex-filter December. Adding reviews { date: 1 }, or storing a parsed year/month field, would convert this into an IXSCAN before the collection grows materially.

Schema and indexes did not change between Stage 2 and Stage 3. No collections were added, no indexes were added, and no key orders were edited. Every per-record lookup in Q1–Q6 uses a Stage-2 index.

12Lessons learned and recommendations
End of deck

Lessons specific to MongoDB

explain proved the most useful tool during development.

  • mongodb-memory-server reduces the design loop from minutes to seconds and is recommended as the default for prototyping.
  • explain('executionStats') was the most valuable design-validation tool; its output should be committed alongside the query code.
  • Effective schema design happens at index-design time, not collection-design time.

Lessons that transferred outside MongoDB

Planner output is the only reliable signal of actual behaviour.

  • Multi-step orchestration in application code is preferable to nested $lookup for both clarity and memory in Q3, Q4, and Q6.
  • Pseudocode is necessary for design, but the query planner output is the only evidence of how the query will actually run.
  • Loader and queries should be developed together; a schema is not validated until data has actually been loaded into it.
Recommended workflow for a similar project
1 · Load before finalising schema

Stand up mongodb-memory-server early; a schema cannot be evaluated against a dataset that has not been loaded.

2 · Add indexes per query, not in advance

Write the query, run explain, and add the index the planner identified. Avoid speculative indexing.

3 · Prefer application code to nested $lookup

When a join requires more than one stage, perform it in application code using indexed point reads. Easier to read and easier to debug.

4 · Commit results and explain output with the source

Each query should have a JSON result file and an explain file committed alongside the implementation, so that behaviour is auditable.
