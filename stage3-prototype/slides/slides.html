07Q3 — Salem entire-home windows
Live demo

Demo · Q3 · App-side scanner

Bookable windows that respect minimum_nights.

  • 1. One indexed find for Salem entire-home listings
  • 2. One sorted find over calendar { city, listing_id, date }
  • 3. Walk consecutive available:true runs; keep day d only if remaining-run-length ≥ minimum_nights[d]
Why app-side? A listing might say available Saturday but require a 7-night minimum. If only Sat–Sun are open you can't actually book it. Mongo can't express this in aggregation alone.
2 indexed reads per listing min-nights aware

10Q6 — Re-book reminder + same-host
Live demo

Demo · Q6 · Touches all 4 collections

For repeat reviewers — was the listing free again?

  • 1. Aggregate reviews → repeat (city, listing, reviewer) groups
  • 2. Per group: findOne listing · findOne calendar · find other host listings
  • 3. Emit enriched record
No new indexes for any of the 6 queries. Every per-record read in step 2 lands on an index already designed in Stage 2 — strong signal the index design generalises.

One aggregation + a few indexed point reads beats a single deeply-nested $lookup for both clarity and memory.

11Critique — what we'd change
Honest postmortem

Things we'd redo, in order of impact

The 4-collection / 8-index core held up well.

01 · Index key order

host_id_1_city_1 is wrong.

Should be { city, host_id } so it shares a city prefix with our other three listings indexes. Affects Q6's same-host lookup.

02 · Date-as-string rationale

Right call, wrong reason.

Stage 2 justified it on storage. The actual win was query simplicity — Q5's regex + $substr with no $toDate cost.

03 · Embed neighborhoods

Per-city array would beat the $lookup.

Eliminates the $lookup in Q2 entirely. A few hundred bytes per city in exchange for one fewer cross-collection hop.

What we did change between Stage 2 and Stage 3: nothing in the schema or indexes. Stage 3 is a true validation of Stage 2, not a redesign.

12Lessons learned
What we'd tell the next team

About MongoDB

explain is the
most useful thing
in the toolbox.

  • mongodb-memory-server shrinks the design loop from minutes to seconds — make it the default for prototyping.
  • explain('executionStats') is the single most valuable design-validation tool — commit its output alongside query code.
  • Schema design happens at index-design time, not collection-design time.

About cloud data systems generally

Pseudocode is a guess.
A plan is proof.

  • Multi-step orchestration in app code beats nested $lookup for clarity and memory (Q3, Q4, Q6).
  • "Use the planner" — pseudocode is guessing, explain is proof.
  • Ship the loader and the queries together; the schema isn't real until something has actually loaded into it.
