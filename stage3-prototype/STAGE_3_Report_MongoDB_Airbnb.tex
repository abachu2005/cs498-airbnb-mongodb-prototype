\section{Stage 3 Goal and Scope}

Stage~2 designed four collections (\texttt{listings}, \texttt{calendar}, \texttt{reviews}, \texttt{neighborhoods}), eight compound indexes, and pseudocode for all six AirBnB queries. Stage~3 turns that design into a working prototype that boots MongoDB via \texttt{mongodb-memory-server}, loads the \emph{full} InsideAirbnb datasets for four cities (Los Angeles, Portland, Salem, San Diego), and runs \textbf{all six queries} end-to-end with \texttt{explain('executionStats')} captured so we show \emph{actual} index usage rather than promised index usage. The schema and indexes are unchanged from Stage~2~\S2.2; everything else---loader, six query modules, driver, web demo UI, evidence---is new in \texttt{stage3-prototype/}. The prototype boots with one \texttt{npm~start} (no external Mongo required) or with \texttt{npm~run~web} for an interactive single-page UI.

\section{Loader Architecture and Data Volume}

\texttt{src/loader.js} streams \texttt{listings.csv.gz}, \texttt{reviews.csv.gz}, \texttt{calendar.csv.gz}, and \texttt{neighbourhoods.csv} from \texttt{data.insideairbnb.com} through \texttt{zlib.createGunzip()} into \texttt{csv-parse} (no full CSV ever held in memory), batches inserts at 2--10\,k rows per call so heap stays bounded, cleans rows (currency strip, \texttt{t/f}\,$\to$\,boolean, neighborhood dedup), and creates the eight Stage~2 indexes after load. Per-collection caps are configurable via \texttt{LIMIT\_*} env vars but default to \emph{none}. Recorded full-data run: \textbf{63{,}482 listings, 3{,}200{,}722 reviews, 481 neighborhoods, 23{,}170{,}957 calendar rows} (LA: 45.6k listings / 16.6M calendar; Portland: 4.5k / 1.6M; Salem: 280 / 102k; San Diego: 13.2k / 4.8M)---the entire InsideAirbnb snapshot for all four cities, no row caps. Total load + index time on a laptop: $\sim$6.5\,minutes. Full per-city counts and a sample-document dump live at \texttt{stage3-prototype/out/load\_evidence.txt}.

\section{Query Implementations and Execution Plans}

All six queries live as standalone modules (\texttt{src/q1.js}\,Q1, \texttt{src/q2.js}\,Q2, \texttt{src/q3.js}\,Q3, \texttt{src/q4.js}\,Q4, \texttt{src/q5.js}\,Q5, \texttt{src/q6.js}\,Q6). The driver runs every query, writes results to \texttt{out/qN\_results.json} and the planner output to \texttt{out/explain\_qN.json}.

\subsection{Q1 -- Portland 2-day search}
\begin{lstlisting}
[ {$match:{city:"portland", date:{$in:dates}, available:true}},
  {$group:{_id:"$listing_id", a:{$sum:1}}}, {$match:{a:dates.length}},
  {$lookup:{from:"listings", let:{l:"$_id"}, as:"b", pipeline:[
     {$match:{$expr:{$and:[{$eq:["$city","portland"]},{$eq:["$listing_id","$$l"]}]}}},
     {$project:{_id:0,name:1,neighborhood:1,room_type:1,accommodates:1,property_type:1,price:1,review_scores_rating:1}}]}},
  {$unwind:"$b"},{$replaceRoot:{newRoot:"$b"}},
  {$sort:{review_scores_rating:-1, listing_id:1}}, {$limit:25} ]
\end{lstlisting}
The driver picks demo dates dynamically (earliest pair of consecutive available Portland dates), so the query always has live data. \textbf{Plan} (full-data run, \texttt{out/explain\_q1.json}): leading \texttt{\$match} hits \texttt{IXSCAN} on \texttt{calendar~\{city,~date,~available\}} with bounds locked to Portland and the two requested dates; \texttt{nReturned=1826}, \texttt{totalKeysExamined=3012}, \texttt{totalDocsExamined=3012}, \texttt{executionTimeMillis=112}---one key+doc per \texttt{(listing,~date)} match across the 1.6M-row Portland calendar partition, then \texttt{\$lookup} hits unique \texttt{listings~\{city,~listing\_id\}} for 1.8k point reads, and \texttt{\$limit:25} caps the output.

\subsection{Q2 -- Empty neighborhoods in a month}
Aggregate \texttt{calendar} for the target city/month with \texttt{available:true}, group by \texttt{listing\_id}, \texttt{\$lookup} each listing's \texttt{neighborhood}, project a set of ``active'' neighborhoods, then anti-join in app code against the full \texttt{neighborhoods} list for that city. Returns the dark neighborhoods (zero available nights). \textbf{Plan:} leading \texttt{\$match} hits \texttt{calendar~\{city,~date,~available\}} with bounds locked to the month range; the lookup hits unique \texttt{listings~\{city,~listing\_id\}}; the final neighborhood pull is a covered scan of \texttt{neighborhoods~\{city,~neighborhood\}}.

\subsection{Q3 -- Salem entire-home availability windows}
For each Salem entire-home listing (one indexed \texttt{find} on \texttt{listings~\{city,~neighborhood,~room\_type\}}), pull the month's calendar rows in one \texttt{find} on \texttt{calendar~\{city,~listing\_id,~date\}} sorted by listing then date. App-side scan turns each contiguous run of \texttt{available:true} days into intervals, keeping only days where \emph{remaining-run-length} $\geq$ that day's \texttt{minimum\_nights}---i.e., we exclude days that look free but are actually unbookable because no valid stay fits. Output: per-listing intervals \{from, to, min\_nights\}.

\subsection{Q4 -- Portland March--August booking trend}
Loops Q3's interval logic over the six target months for Portland entire-home listings and sums each month's bookable nights. Output: \texttt{[\{month,~listings\_with\_intervals,~total\_bookable\_nights\}]}. Same indexes as Q3; six passes over the same access path.

\subsection{Q5 -- December reviews per city per year}
\begin{lstlisting}
[ {$match: {date: {$regex: /^\d{4}-12-\d{2}$/}}},
  {$group: {_id:{city:"$city", year:{$substr:["$date",0,4]}}, review_count:{$sum:1}}},
  {$project:{_id:0, city:"$_id.city", year:"$_id.year", review_count:1}},
  {$sort:{city:1, year:1}} ]
\end{lstlisting}
Pure full-collection aggregation by design---every December review must be counted, no useful index prefix applies. \textbf{Plan} (\texttt{out/explain\_q5.json}): \texttt{COLLSCAN} over all 3.2\,M reviews, \texttt{nReturned=193{,}340} matching docs, \texttt{executionTimeMillis=1{,}775}---i.e., $\sim$1.8\,s to scan and group the entire reviews collection on a laptop. The Stage~2 design intentionally omits a leading \texttt{\{date:1\}} on \texttt{reviews}; Q5 is the only date-driven \texttt{reviews} query, and 1.8\,s/3.2\,M docs is well within demo budget. If \texttt{reviews} grew to 100M+ documents, adding \texttt{reviews~\{date:1\}} would be the first tuning step.

\subsection{Q6 -- Re-book reminder + same-host listings}
\textbf{Step~1} (one aggregation): group \texttt{reviews} by \texttt{(city, listing\_id, reviewer\_id)}, keep \texttt{review\_count$>$1}. \textbf{Step~2} per group (capped at 25 for the demo): one \texttt{findOne} on \texttt{calendar~\{city,~listing\_id,~date\}} for in-month availability, one aggregate over the same index for min/max nights, one \texttt{findOne} on \texttt{listings~\{city,~listing\_id\}} for description and host metadata, one \texttt{find} on \texttt{listings~\{host\_id,~city\}} for the host's other listings in the same city. Every step-2 read lands on an index already designed in Stage~2---\textbf{Q6 needed no new indexes}, validating the design.

\section{Critique and Lessons}

\textbf{(1) Date-as-string was right but for the wrong stated reason.} Stage~2 justified ISO-string dates on storage. The bigger payoff is \emph{query simplicity}: Q5's \texttt{\$substr} is regex-prefix-indexable, Q3/Q4's calendar walk is just string comparison, whereas \texttt{\$year}/\texttt{\$month} on a true \texttt{Date} forces a per-document compute that no index can short-circuit. \textbf{(2) The \texttt{host\_id\_1\_city\_1} index has the wrong key order.} Reversing it to \texttt{\{city, host\_id\}} would share a \texttt{city} prefix with our three other listings indexes and be more consistent. \textbf{(3) \texttt{neighborhoods} could be embedded as one document per city}; Q2 would then skip the \texttt{\$lookup} entirely.

\textbf{Lessons.} \texttt{mongodb-memory-server} shrinks the design loop to seconds; it is the right default for prototyping. \texttt{explain('executionStats')} is the single most valuable tool we used---one line of \texttt{.explain()} per query turns guessed index usage into proven index usage. Multi-step orchestration in app code (Q6, Q3, Q4) is meaningfully easier to read and debug than deeply-nested \texttt{\$lookup} pipelines, and at our scale costs essentially nothing because every read hits an index. \textbf{Advice.} Pick one query first and design every collection and every index around it; only then add the next. Doing all six in one pass tempts you into ``just-in-case'' indexes (we did this with \texttt{host\_id\_1\_city\_1}). Always commit \texttt{explain} output alongside results so reviewers (and future you) can verify what the planner is actually doing.

\section{My Role in the Project}

I led infrastructure, data ingestion, and demo UI for the prototype. Concretely: I scaffolded \texttt{stage3-prototype/}, picked \texttt{mongodb-memory-server} so the team could iterate on queries without an Atlas account, and wrote the loader (\texttt{src/loader.js})---streaming gzipped CSVs from \texttt{data.insideairbnb.com} through \texttt{zlib} and \texttt{csv-parse}, batching inserts so the full multi-million-row load fits in memory, normalizing types, deduplicating neighborhoods to satisfy the unique index, and creating all eight Stage~2 indexes. I also wrote the driver (\texttt{src/driver.js}) that boots in-process Mongo, calls the loader, runs every query end-to-end, captures \texttt{explain('executionStats')} for each, and writes everything in \texttt{out/} for reproducibility. On the query side I implemented Q1 (Portland 2-day search) including the dynamic date-picking helper that finds the earliest pair of consecutive available Portland dates so the demo always returns live results. I built the demo UI (\texttt{web/}, vanilla HTML/CSS/JS over an Express server that wraps the same query modules so the report and the UI run identical code), and I owned the GitHub repository (\texttt{cs498-airbnb-mongodb-prototype}), the top-level \texttt{README}, and the PDF generation pipeline (\texttt{tectonic} for the report, \texttt{md-to-pdf} for the slide outline). The biggest decision I argued for was \emph{not} setting up Atlas in week~1: the in-process server let us focus on schema and query correctness from day one, and ``switch to Atlas'' is one URI change away. I integrated my teammates' query implementations into the driver and the web UI, and re-ran the evidence pipeline whenever the schema or query logic changed.
